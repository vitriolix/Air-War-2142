# `design/` — Code ↔ Claude Design round-trip

This folder is the handoff workspace between the **codebase** (Kotlin/KorGE UI) and
**[Claude Design](https://claude.ai/design)**, where we mock up and refine the UI as HTML/CSS
components before porting changes back into the scenes.

**The game:** Air War 2142 is an **alt-history reimagining** of the WWII air-war shmup — not a
1942 clone. In its timeline WWII-style warfare never gave way to jets/missiles; it kept evolving
for two centuries into a fusion of eras (riveted prop-planes + energy weapons + sci-fi HUDs),
steampunk-style but anchored to WWII. See `PROMPT.md` §1 for the full theme — it's the direction
to refine *toward*, not what the current screens show.

This bundle is the **shared UI spec for both sides**: Claude Design refines against it, and
Claude Code ports changes back into the Kotlin scenes from it. One source of truth, not two.

### How this relates to the official Claude Design handoff

The official flow (verified June 2026) is **Design → Code, one-directional**: Claude Design
exports a bundle (HTML/CSS/JS + per-state screenshots + a README) that Claude Code picks up,
and iteration happens through **GitHub PR review** — not a semantic design↔code diff. There's
no official data-model-ownership split or live agent↔agent Q&A; the recommended substitute is a
README that *preempts* questions. We keep this custom `design/` layer because the official path
emits web HTML and this is a **KorGE/Kotlin** target — there's no HTML component to hand back,
so the bundle must be hand-ported into the scenes. `design-tokens.json` (contract) + `spec.html`
(real screenshots) + that port is our non-web adaptation; transport can still use the official
`DesignSync` / `/design-sync`. (Details + the banked correction: `design_code_handoff_official`
project memory.)

## Gradle tasks (group `game`)

| Task | Direction | What it does |
|------|-----------|--------------|
| `./gradlew designExport` | **Code → Design** | Refreshes `assets/` from the committed sprite atlas, validates the bundle (`PROMPT.md`, `design-tokens.json`, `spec.html`), opens `spec.html`, and prints how to push to Claude Design. |
| `./gradlew designImport` | **Design → Code** | Scans `incoming/`, lists what came back, opens any HTML for review, and prints how to apply changes (update `design-tokens.json` first, then the scene that owns each value). |
| `./gradlew captureScreens` | refresh baseline | Recaptures `screens/*.png` from the live web build — **headed + 1:1 only** (headless SwiftShader fakes SDF text slivers; see `scripts/capture-screens.js`). |
| `./gradlew renderDocs` | publish spec | Renders the docs site (this README + the linked spec artifacts) to `build/docs/` and opens it. |

`npm run design:export` / `design:import` / `design:capture` are thin shims for the same tasks.

## Workflow

**Code → Design.** Run `./gradlew designExport`. It stages the bundle and opens the spec. Then open a
Claude Design project, paste `PROMPT.md`, and attach `design-tokens.json`, `spec.html` (+ `screens/`),
`refinement-mockup.html`, and `assets/sprites.png` + `assets/sprites.txt`. (If you have the
`/design-sync` skill, it can push `design/**` to your claude.ai/design project for you.)

**At each release.** Snapshot the current UI/design state in [`DESIGN_HISTORY.md`](DESIGN_HISTORY.md) (newest
entry at the bottom) + a committed bundle copy under `design/design-history/<id>/`. Cadence is per *release*, not
per Design exchange — a release may fold in many round trips, which all roll up into one entry.

**Design → Code.** Pull the refined component files into `design/incoming/` (download from the project, or
let `/design-sync` write them there), then run `./gradlew designImport` to review. Apply changes by:
1. updating `design-tokens.json` for any changed/added token (it's the contract), then
2. editing the scene that owns the value (`MenuScene.kt`, `GameScene.kt`, `SettingsScene.kt`).

Finally **verify by running the app** (per the project's hard rules) — a design change isn't done
until it renders correctly in the real game.

## Source-of-truth rules

- `design-tokens.json` is authoritative. `spec.html` is the faithful current-state baseline (real
  screenshots + real sprites); `refinement-mockup.html` is aspirational and must never be read as
  current truth. When anything disagrees, `design-tokens.json` + the Kotlin source win. `PROMPT.md`
  §5 lists the exact strings/sizes/positions per screen.
- `assets/` and `incoming/` are derived/scratch and gitignored — never hand-edit `assets/`; re-run
  `designExport`. The committed atlas lives at `composeApp/src/commonMain/resources/sprites.{png,txt}`.
