- [ ] **Web canvas sizing / "Chrome for Testing" renders huge vs `playWeb` — two KorGE behaviors, neither a rendering bug (deferred, josh — revisit later).** Symptom: the game window is huge in the Playwright "Chrome for Testing" browser (capture/probe runs) but normal in `playWeb`. Investigated + probed the live JS build; the "2× / huge" has **two distinct, separable causes** — don't conflate them:
  - **Window size = the visible symptom (KorGE fills-and-scales).** KorGE's JS canvas is *responsive*: it fills whatever window it's in — probed `canvas CSS == window` at 1440×900, 800×700, and 1000×1500 — then scales the **virtual 1000×1500** game to fit (letterboxed). So in a normal browser window (shorter than 1500px tall) the portrait game scales **down** → looks normal; but `scripts/capture-screens.js` + the probes **pin the Playwright viewport to 1000×1500** (`deviceScaleFactor:1`) to get pixel-exact 1000×1500 baseline PNGs, so KorGE renders full native 1:1 and the window is physically 1500px tall = huge on a laptop. The screenshots it produces are still correct 1000×1500. **Not a bug — the capture viewport choice meeting KorGE's fill-the-window behavior.**
  - **Backing-buffer DPR = invisible, but a perf blind-spot.** KorGE sizes the WebGL drawing buffer to `cssPx × devicePixelRatio` (probed: 2000×3000 backing on a `deviceScaleFactor:2` retina context, CSS still 1000×1500). `main.kt`'s `Korge(...)` never sets `GameWindow.Quality`, so it rides KorGE's default (renders at DPR). On retina that's **~4× the fragment/fill work per frame**. The perf-gate baseline (web median ~3.9ms — see Constraints) was measured headed at Playwright's default `deviceScaleFactor:1` (DPR 1), so **a real retina user renders at 4× the pixels and that cost is currently unmeasured** — a latent perf hole vs the gate.
  - **Deferred options (no action taken):** (1) shrink the capture window without changing output — capture at viewport **500×750 @ `deviceScaleFactor:2`** → still a 1000×1500 PNG, integer-clean (the SDF slivers were *software-GL* only, so DSF=2 on real GPU is fine), but the on-screen window is only ~500×750; (2) set `GameWindow.Quality` explicitly in `Korge(...)` — `PERFORMANCE` forces 1× backing (cheaper, softer on retina) vs the default DPR (crisp, 4× fill); (3) **measure** frame time headed at DSF=2 vs `PERFORMANCE` (LPM off) to size the retina cost before deciding. Related: the `check_low_power_mode` memory (LPM throttling is a *perf* gate only, not a visual one) and the headed-vs-headless SDF software-GL note in Done below.

- [x] **"Text-rendering artifact" was a software-GL rendering artifact in the capture tooling, NOT a font/game/KorGE bug — resolved 2026-06-16.** Thin vertical slivers ("FLIGHTi CONTiROLiS", "STiEERiING") appeared ONLY in `scripts/capture-screens.js` output, never in the real game. **Isolated root cause:** rendering KorGE's SDF text under **SwiftShader (software GL)** — i.e. headless Chromium — not the font, the glyphs, or the screen scale. KorGE's SDF anti-aliasing (`SDFShaders.opAARev` = `clamp(d/fwidth(d)+0.5)`) derives the AA width from screen-space derivatives (`fwidth`); SwiftShader computes those differently from hardware GL, and on ~1-texel-thin stems the threshold clips → slivers. (The default font is *Sani Trixie Sans* via `DefaultTtfFontAsBitmap = DefaultTtfFont.lazyBitmapSDF`, but the font is incidental — any SDF text shows it under software GL.) **Variable isolation:** real GPU @ integer 1:1 = clean; SwiftShader @ integer 1:1 = slivered; SwiftShader @ old 1.2× = slivered. So the non-integer resample (old 600×900@DSF2 capture) was a **red herring**, not a cause — software GL alone does it. *Cause was mis-called three times before this isolation (font glyphs → "SDF path in-game" → "software GL + resample"); the game was always fine.* **Not a KorGE defect:** the AA shader is the standard textbook technique and renders perfectly on real GPUs — the artifact is SwiftShader's `fwidth` precision (environment limitation). At most a low-priority upstream *robustness enhancement* (derivative-free AA width from a texel-size uniform); not worth filing. **No font swap / no SDF change needed.** **Fix landed:** `capture-screens.js` defaults to headed + 1:1 native 1000×1500; `HEADLESS=1 DSF=1 W=1000 H=1500` reproduces the slivers under software GL (that path is for non-visual smoke checks only); `design/screens/*.png` re-captured clean. - [~] #25 **Two-tier seed reproducibility / content + kernel versioning** *(design direction, josh)* — separate **layout stability** (seed → same world layout; needs ONLY `genVersion`, immune to balance/prop edits because worldgen never reads object props) from **full-fidelity replay** (seed plays out identically; coupled to *all* behavior + assets). **Version PER OBJECT, not as a global set** (a single global ruleset version can't work once objects are user-definable): each object carries its own version via its **plugin semver manifest** (plugin-arch #5); a run records the **sparse set of object-versions it touched** (content-addressed for user objects — git holds history, the runtime question is which versions are bundled/executable). Rule: **any sim-affecting change ⇒ MAJOR bump**; minor/patch must be sim-neutral (cosmetic/perf). **"Run old seeds forever" mechanism = a dependency-free, frozen, tiny kernel — NOT archived whole engines** (archiving engines is intractable: every old engine must keep compiling vs future Kotlin/KorGE = dependency-freeze hell). The kernel (ECS tick, RNG, collision/math) **depends on nothing** (pure Kotlin, no korlibs — same no-deps discipline already applied to `worldgen`/`ecs`, which is the precondition that makes this feasible) with a **permanent ABI**, behind a **stable host boundary** `SimRuntime { tick(inputs) → FrameState }` so KorGE/render types never leak in; all behavior lives in versioned plugins, so an old seed loads its plugin versions on the *unchanged* kernel. Replay record carries `(genVersion, kernelAbiVersion, {objectId→version}, seed)`; add a **golden replay-conformance test per archived version** (same pattern as the cross-backend goldens) to catch a future change that breaks an old version. **Seed sharing = a share-code CODEC, not a hash** (a hash is one-way → can't *select* versions): reversible **Crockford base32** encoding a *compact tuple* `(formatVersion, genVersion, rulesetVersion, seed)` — NOT the full lockfile — plus a one-way **check digit** for typo/corruption; two forms = plain seed number (current version implied) vs full code (versions pinned); codec is pure-int → cross-platform clean; version the code format itself. Bundle only explicitly-supported old versions (events/ladders), not infinitely. See `deterministic-worldgen-seeding` + `plugin-architecture` memories; siblings #19 (replay) / #22 (determinism guard).
