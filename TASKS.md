- [ ] **`BLENDER_HANDOFF.md`: clarify rendering ownership + investigate a 90s-style degraded render look.** Today's doc already assigns texturing/paint/decals/lighting/camera/batch-render to Blender (Phases 3–4) and bake-to-atlas to the separate `resvg` pipeline — but this split (Blender = pixels, resvg = packing only, nothing does stylization degradation) isn't stated explicitly anywhere, so make it explicit. Then investigate a retro/90s-style degraded look for the rendered output: hard/jagged edges (no anti-aliasing — Blender render samples=1 or a post-pass edge threshold), pixelation (render at a low native resolution, e.g. 64–128px, and nearest-neighbor scale — never soften/resample), and a limited/quantized color palette (per-biome palette from `PAINT_SCHEMES.md`, k-means or fixed-palette quantization on the render output). Produce one test render of the Bf-109 at each candidate (Blender-baked, post-process-baked, KorGE-runtime) and compare against the existing 2D sprite art's pixel-art style (`render/Shapes.kt`) for consistency and perf.
- [ ] **3D render pipeline: banked-turn + barrel-roll + loop-de-loop animation frames.** The pipeline as documented only produces **static** per-variant renders (chonkyness/stretch/damage/livery — `BLENDER_HANDOFF.md` Phase 4); there's no animated frame-sequence output. Add animation support: (1) a **flat→banked transition** — smooth in-between frames from level flight to a banked pose, used continuously while turning (bank left when moving left, bank right when moving right; return to flat when input is neutral); (2) a **barrel roll** — the aircraft does one full 360° rotation around its own *longitudinal* (nose-to-tail/roll) axis while continuing forward, i.e. it spins like a rifled bullet in flight — as the primary dodge-action animation; (3) a **loop-de-loop** — the aircraft pitches up into a full vertical circle (nose up → inverted at the top → nose down → level again), rotating around its *lateral* (wingtip-to-wingtip/pitch) axis instead of the roll axis — as an alternate dodge animation. **Alt-dodge selection is random, not player-chosen:** each dodge trigger picks barrel-roll or loop-de-loop by chance (weighting/probability TBD), not a new input/power-up/unlock. **Existing hook — not a real barrel roll today:** the 2D vector pipeline has a `roll` mechanic (`GameEngine.triggerRoll()`/`rollsLeft`/`rollProgress`, ~0.8s invulnerability window), but its visual (`GameScene.kt`: `playerView.rotation = Angle.fromDegrees(rollProgress * 360)` + a sine squash on `scaleX`) is a flat **X-Y screen-plane spin** (rotation around the axis pointing out of the screen, like a pinwheel) with a squash/stretch fake, not a true barrel roll (rotation around the aircraft's *longitudinal* axis, which would foreshorten the wings/fuselage through the roll). The code comment in `render/Shapes.kt` explicitly defers "the 3/4 bank pose and every animated FX" — this task is what fulfills that deferral, replacing the screen-plane-spin placeholder with real rendered roll/bank frames once the 3D pipeline exists. **Scope:** camera/rig setup for a rotation sequence in Blender, frame count vs. smoothness/atlas-size tradeoff (perf gate — keep frame counts modest, this bakes to a sprite-atlas animation like any other pre-baked sprite per architecture decision #8), and wiring the resulting frame sequence into `GameEngine`/`GameScene` playback (replacing the current static `PLAYER_ROLL` swap and adding bank state to normal flight). Depends on the base stylization pipeline (this doc's Phases 1–3) landing first for at least one aircraft.

- [ ] **Web canvas sizing / "Chrome for Testing" renders huge vs `playWeb` — two KorGE behaviors, neither a rendering bug (deferred, josh — revisit later).** Symptom: the game window is huge in the Playwright "Chrome for Testing" browser (capture/probe runs) but normal in `playWeb`. Investigated + probed the live JS build; the "2× / huge" has **two distinct, separable causes** — don't conflate them:
  - **Window size = the visible symptom (KorGE fills-and-scales).** KorGE's JS canvas is *responsive*: it fills whatever window it's in — probed `canvas CSS == window` at 1440×900, 800×700, and 1000×1500 — then scales the **virtual 1000×1500** game to fit (letterboxed). So in a normal browser window (shorter than 1500px tall) the portrait game scales **down** → looks normal; but `scripts/capture-screens.js` + the probes **pin the Playwright viewport to 1000×1500** (`deviceScaleFactor:1`) to get pixel-exact 1000×1500 baseline PNGs, so KorGE renders full native 1:1 and the window is physically 1500px tall = huge on a laptop. The screenshots it produces are still correct 1000×1500. **Not a bug — the capture viewport choice meeting KorGE's fill-the-window behavior.**
  - **Backing-buffer DPR = invisible, but a perf blind-spot.** KorGE sizes the WebGL drawing buffer to `cssPx × devicePixelRatio` (probed: 2000×3000 backing on a `deviceScaleFactor:2` retina context, CSS still 1000×1500). `main.kt`'s `Korge(...)` never sets `GameWindow.Quality`, so it rides KorGE's default (renders at DPR). On retina that's **~4× the fragment/fill work per frame**.
