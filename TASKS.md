Investigated + probed the live JS build; the "2√ó / huge" has **two distinct, separable causes** ‚Äî don't conflate them:
  - **Window size = the visible symptom (KorGE fills-and-scales).** KorGE's JS canvas is *responsive*: it fills whatever window it's in ‚Äî probed `canvas CSS == window` at 1440√ó900, 800√ó700, and 1000√ó1500 ‚Äî then scales the **virtual 1000√ó1500** game to fit (letterboxed). The screenshots it produces are still correct 1000√ó1500. **Not a bug ‚Äî the capture viewport choice meeting KorGE's fill-the-window behavior.**

The perf-gate baseline (web median ~3.9ms ‚Äî see Constraints) was measured headed at Playwright's default `deviceScaleFactor:1` (DPR 1), so **a real retina user renders at 4√ó the pixels and that cost is currently unmeasured** ‚Äî a latent perf hole vs the gate.
  - **Deferred options (no action taken):** (1) shrink the capture window without changing output ‚Äî capture at viewport **500√ó750 @ `deviceScaleFactor:2`** ‚Üí still a 1000√ó1500 PNG, integer-clean (the SDF slivers were *software-GL* only, so DSF=2 on real GPU is fine), but the on-screen window is only ~500√ó750; (2) set `GameWindow.Quality` explicitly in `Korge(...)` ‚Äî `PERFORMANCE` forces 1√ó backing (cheaper, softer on retina) vs the default DPR (crisp, 4√ó fill); (3) **measure** frame time headed at DSF=2 vs `PERFORMANCE` (LPM off) to size the retina cost before deciding. - [x] **"Text-rendering artifact" was a software-GL rendering artifact in the capture tooling, NOT a font/game/KorGE bug ‚Äî resolved 2026-06-16.** Thin vertical slivers ("FLIGHTi CONTiROLiS", "STiEERiING") appeared ONLY in `scripts/capture-screens.js` output, never in the real game. **Isolated root cause:** rendering KorGE's SDF text under **SwiftShader (software GL)** ‚Äî i.e. headless Chromium ‚Äî not the font, the glyphs, or the screen scale. KorGE's SDF anti-aliasing (`SDFShaders.opAARev` = `clamp(d/fwidth(d)+0.5)`) derives the AA width from screen-space derivatives (`fwidth`); SwiftShader computes those differently from hardware GL, and on ~1-texel-thin stems the threshold clips ‚Üí slivers. (The default font is *Sani Trixie Sans* via `DefaultTtfFontAsBitmap = DefaultTtfFont.lazyBitmapSDF`, but the font is incidental ‚Äî any SDF text shows it under software GL.) **Variable isolation:** real GPU @ integer 1:1 = clean; SwiftShader @ integer 1:1 = slivered; SwiftShader @ old 1.2√ó = slivered. So the non-integer resample (old 600√ó900@DSF2 capture) was a **red herring**, not a cause ‚Äî software GL alone does it. *Cause was mis-called three times before this isolation (font glyphs ‚Üí "SDF path in-game" ‚Üí "software GL + resample"); the game was always fine.* **Not a KorGE defect:** the AA shader is the standard textbook technique and renders perfectly on real GPUs ‚Äî the artifact is SwiftShader's `fwidth` precision (environment limitation). At most a low-priority upstream *robustness enhancement* (derivative-free AA width from a texel-size uniform); not worth filing. **No font swap / no SDF change needed.** **Fix landed:** `capture-screens.js` defaults to headed + 1:1 native 1000√ó1500; `HEADLESS=1 DSF=1 W=1000 H=1500` reproduces the slivers under software GL (that path is for non-visual smoke checks only); `design/screens/*.png` re-captured clean. (Both via the companion design-round-trip PR.) **Note:** the earlier "Verified: reproduces in-game" claim was wrong ‚Äî judged from headless screenshots, not the live headed app.
